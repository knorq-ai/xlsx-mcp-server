# Token Efficiency Analysis: MCP Tools vs Raw Python

xlsx-mcp-server の MCP ツールと、エージェントが直接 Python コードを生成・実行する場合のトークン消費量を比較する。

## 前提

- **トークン換算**: 英語テキスト 1 token ≈ 4 文字 / JSON 1 token ≈ 3–4 文字
- **Python アプローチ**: エージェントが openpyxl コードを生成 → Bash で実行 → 結果を解析
- **MCP アプローチ**: 構造化パラメータでツールを呼び出し → 構造化レスポンスを受信
- **コスト基準**: Claude Sonnet 入力 $3 / 出力 $15 per 1M tokens (2025年時点)

## 一次コスト: ツールスキーマ (MCP のみ)

MCP サーバ接続時にツール定義がコンテキストに追加される。

| 項目 | トークン数 |
|------|-----------|
| 37 ツール × スキーマ (名前 + 説明 + パラメータ型) | **~2,700** |
| サーバ説明 + 制限事項テキスト | **~200** |
| **合計 (1 セッション 1 回のみ)** | **~2,900** |

このコストは 1 セッションで 1 回だけ発生し、複数回のツール呼び出しで償却される。

## シナリオ別比較

### シナリオ 1: シートの内容を読む

**タスク**: 出勤簿.xlsx のシート「出勤簿」の A1:G20 を読み取る。

#### MCP アプローチ

| ステップ | 入力トークン | 出力トークン |
|---------|-------------|-------------|
| ツール呼び出し `read_sheet(file_path, "出勤簿", "A1:G20")` | ~50 | ~30 (tool_use) |
| レスポンス受信 (20行 × 7列 + JSON) | ~400 (tool_result) | — |
| **合計** | **~450** | **~30** |

#### Python アプローチ

| ステップ | 入力トークン | 出力トークン |
|---------|-------------|-------------|
| Python コード生成 (import, open, iterate, print) | — | ~200 |
| Bash 実行コマンド | ~20 | ~30 |
| 実行結果の読み取り | ~300 | — |
| 結果の解釈・要約をユーザに返す | — | ~100 |
| **合計** | **~320** | **~330** |

**差分**: MCP は出力 **-300 tokens** (出力は入力の 5 倍の単価のため、コスト換算で大きな差)

### Python アプローチのデバッグ反復コスト (未計上)

上記の Python 推定値は **1 回で正しいコードが生成される前提** である。実際には:

- openpyxl の API を間違える (例: `PatternFill` の引数、`Border` のコンストラクタ)
- ファイルパスやシート名の encoding エラー (日本語パスなど)
- import 漏れ、型エラー

各反復で追加 **~300-500 出力トークン** が発生する。1 タスクで平均 1.5 回の反復が発生する場合、Python の実質コストはさらに **50%** 増加する。

## 損益分岐点

MCP の一次コスト (ツールスキーマ ~2,900 tokens) を回収するのに必要な操作回数:

- 1 操作あたりの平均出力削減: ~300 tokens
- 出力 300 tokens × $15/1M = $0.0045 / 操作
- スキーマコスト: 2,900 tokens × $3/1M = $0.0087 (入力単価)
- **損益分岐: 2 操作** (3 回目以降は純粋にコスト削減)

実務では 1 セッションで最低 3-5 回の Excel 操作を行うため、**初回から黒字** になるケースが大半である。

## まとめ

| 指標 | MCP ツール | Raw Python |
|------|-----------|------------|
| 出力トークン削減 | **82-91% 削減** | ベースライン |
| コスト削減 | **69-83% 削減** | ベースライン |
| デバッグ反復 | なし (バリデーション済み) | 平均 1.5 回/タスク |
| 損益分岐 | **2 操作** | — |
| エージェント認知負荷 | 低 (宣言的パラメータ) | 高 (API 記憶 + コード生成) |

**結論**: MCP ツールは主に **出力トークンの大幅削減** によりコスト効率が良い。出力トークンは入力トークンの 5 倍の単価であるため、コード生成を省略できる MCP の優位性は大きい。特に書式設定のような API が冗長な操作で効果が顕著である。
